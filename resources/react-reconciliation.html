Reconciliation, Fiber Architecture, and Concurrency Explained

Reconciliation Diffing Stack vs Fiber Event Loop Lanes

1. What is Reconciliation?

In React, you don't manipulate the DOM directly. Instead, you describe the state of your UI. React takes that description and ensures the DOM matches it.

Reconciliation is the process where React compares the new "Virtual DOM" tree with the previous one to calculate the minimal set of changes required.

2. The Diffing Algorithm

Comparing two trees has a complexity of O(n³). To keep things fast, React uses a heuristic O(n) algorithm based on two assumptions:

  1. Different Types: Two elements of different types will produce different trees. React will tear down the old tree and build the new one from scratch.
  2. Keys: Developers can hint which elements are stable across renders with a key prop.

The Logic Flow:

  • If Type changes (e.g., <div> to <span>) → Unmount & Remount.
  • If Type is the same but Props change → Update the underlying instance.
  • When diffing children → React iterates over both lists and matches them by index or key.

3. React 15: The Stack Reconciler

React 15 used a synchronous recursive approach. It followed the call stack.

The "Stack" Problem:

Once React started reconciling, the main thread was locked. If you had 2,000 components to diff, and it took 200ms, the user couldn't click anything during those 200ms. Frames were dropped, and the UI felt frozen.

4. React 16+: The Fiber Architecture

Fiber is a reimplementation of the stack specifically for React components. You can think of a Fiber as a virtual stack frame.

The Anatomy of a Fiber Node

Unlike React 15's internal elements, a Fiber is a persistent object that holds:

type: The component type (div, MyComponent)
key: The unique identifier
child: Pointer to the first child
sibling: Pointer to the next sibling
return: Pointer to the parent (where to return after work)
pendingProps: Props from the new element
memoizedState: The actual state (hooks are stored here in a linked list)
lanes: Priority level of this update

5. Integration with the Event Loop

The "Magic" of Fiber is that it can pause work. It does this by leveraging the Browser's Event Loop.

The 16ms Rule

To keep 60 FPS, a frame must be rendered every 16.6ms. If JS runs for 30ms, the frame is skipped.

How React Yields:

  1. React starts a "Work Loop".
  2. After processing each Fiber, it checks: shouldYieldToHost().
  3. If the 5ms time slice is up, React pauses, saves its place (the current Fiber pointer), and schedules a new task on the browser's message queue.
  4. The browser handles clicks/paints, then gives control back to React to resume.

6. Priority Lanes: The Bitmask Engine

React 18 replaced "Expiration Times" with Lanes. While the old system was linear, Lanes are multi-dimensional. This allows React to work on multiple updates at once without them blocking each other.

A. How the Bitmask Works

React uses a 32-bit integer to represent lanes. Each bit is a "lane." There are 31 available lanes (the 32nd bit is reserved). Lower bits have higher priority.


// High Priority (Sync)
0b0000000000000000000000000000001 

// Low Priority (Transition)
0b0000000000000000000000001000000

Because it's a bitmask, React can check for multiple priorities at once using the & (AND) operator. If a bit is "on," work is pending in that lane.

B. Parallel Worlds (Interleaving)

This is the most powerful feature. Imagine you have a SyncLane update (typing in a box) and a TransitionLane update (filtering a list of 10,000 items).

  • In the old system, the list filter would have to finish before the typing could show up.
  • With Lanes, React can pause the "List Filter" lane, jump over to the "Typing" lane, commit it to the DOM, and then jump back to the list filter.
  • If the typing changed the filter parameters, React simply discards the old list filter work and starts a new one. This is "Parallel Worlds" in action.

C. Lane Expiration (Starvation Prevention)

What if a low-priority update (like a background data fetch) is constantly interrupted by high-priority clicks? It would never finish. This is called Starvation.

The Fix: Every lane has an Expiration Time. When a lane is created, React calculates a timestamp (e.g., 5 seconds in the future). If that time passes and the work still isn't done, React promotes that lane to a SyncLane. It then forces the update to complete synchronously, ensuring the UI eventually stays consistent.

D. Entanglements

Sometimes updates in different lanes depend on each other. React "entangles" them so they commit at the exact same time. This prevents "UI Tearing"—where part of the screen shows new data and another part shows old data.

7. Expert Deep Dive: The Inner Mechanics

A. Double Buffering (The "WIP" Tree)

React uses a technique called Double Buffering to prevent users from seeing partially rendered UI. It maintains two trees:

  • Current Tree: What is currently on the screen.
  • Work-In-Progress (WIP) Tree: The tree being built during the Render phase.

Once the WIP tree is complete, React simply swaps the pointer (root.current = wip) during the Commit phase. This is atomic and instantaneous.

B. The Lifecycle of a Unit of Work

Every Fiber goes through two main functions in the Work Loop:

1. beginWork(): "Going down" the tree. React checks if the component needs an update. If it doesn't (due to memo or unchanged props), it "bails out" and skips the subtree.

2. completeWork(): "Coming up" the tree. After children are processed, React builds the physical DOM nodes (if they don't exist) and bubbles up "Effect Tags" (e.g., "this whole subtree has a deletion").

C. Scheduler vs. Reconciler

Expert-level understanding requires distinguishing these two packages:

Package Responsibility Analogy
Scheduler Determines when work should happen based on priority. The Traffic Controller
Reconciler Determines what needs to change (Diffing). The Construction Worker

8. The Scheduler: The Engine of Concurrency

You might wonder: "How does React actually pause JS?" It doesn't use requestIdleCallback (too inconsistent). Instead, it uses a MessageChannel.

The MessageChannel Trick

  1. React starts work.
  2. After each fiber, it checks if 5ms have passed.
  3. If yes, it creates a MessageChannel and sends a message to itself: port.postMessage(null).
  4. This message goes into the Browser's Task Queue (Macro-task).
  5. The browser finishes its paint/input handling, then sees the message and gives control back to React.

The `alternate` Pointer: The Secret to Resuming

Every Fiber node has an alternate property. This is the link between the Current (on-screen) and Work-In-Progress (building) versions of a component.

            graph LR
                C[Current Fiber] -- alternate --> W[WIP Fiber]
                W -- alternate --> C
                style C fill:#4d79ff,color:#fff
                style W fill:#f96,color:#fff

When React pauses, it simply stores a reference to the "Current WIP Fiber." When it resumes, it follows the alternate pointer to see what the previous state was and continues from there. This "Clone-on-Update" strategy ensures that if an update is aborted, the original "Current" tree remains untouched and stable.

9. The 3 Steps of the Commit Phase

Once the WIP tree is complete, React enters the Commit Phase. This is synchronous and divided into three sub-phases:

  • Phase 1: Before Mutation: React calls getSnapshotBeforeUpdate. The DOM hasn't changed yet.
  • Phase 2: Mutation: This is where the swap happens. React traverses the effect list and performs DOM operations (append, remove, update text). The root.current pointer is swapped here.
  • Phase 3: Layout: The DOM is updated, but the browser hasn't painted yet. React calls useLayoutEffect and componentDidMount.

10. The Grand Unified Trace: From Click to Screen

This is the "Full Stack" view of a React update. Follow the numbers:

Step 1: The Browser Event

You click a button. The Browser puts a "Click Event" into its Task Queue.

Step 2: React's Synthetic Event System

React doesn't attach listeners to every button. It attaches one listener to the Root DOM container. When the click bubbles up:

  • React intercepts it and creates a SyntheticEvent.
  • It looks at the nativeEvent.target and finds the corresponding Fiber Node (React stores this as a property on the DOM node).
  • It finds your onClick handler in that Fiber's memoizedProps.

Step 3: Lane Assignment

Before running your code, React asks the Scheduler: "What is the priority of this user interaction?"

  • Since it's a click, it assigns the SyncLane (highest priority).
  • If it were a mousemove, it might assign a lower priority lane.

Step 4: Scheduling the Work

React calls scheduleUpdateOnFiber. It marks the Fiber and all its ancestors with the new Lane (dirty checking). Then it tells the Scheduler: "I have work for Root X at SyncLane priority."

Step 5: The Concurrent Work Loop (The "Heart")

The Scheduler eventually calls React's workLoopConcurrent. This is where the yielding happens. Here is the actual logic in pseudo-code:


function workLoopConcurrent() {
  // Perform work until we run out of fibers OR the browser needs the thread
  while (workInProgress !== null && !shouldYield()) {
    performUnitOfWork(workInProgress); 
  }
}

function performUnitOfWork(unitOfWork) {
  const current = unitOfWork.alternate;
  // 1. beginWork: Diffing happens here. Returns the 'child' Fiber.
  let next = beginWork(current, unitOfWork, renderLanes);
  
  unitOfWork.memoizedProps = unitOfWork.pendingProps;
  if (next === null) {
    // 2. No more children? Move to siblings or complete this branch
    completeUnitOfWork(unitOfWork);
  } else {
    workInProgress = next;
  }
}

Step 6: Pausing and Resuming

If shouldYield() returns true (meaning the 5ms is up):

  • The while loop breaks.
  • React remembers the current workInProgress fiber pointer.
  • It tells the Scheduler: "I'm not done, call me back when the browser is free."
  • The browser paints. Then the Scheduler calls workLoopConcurrent again.
  • Because workInProgress was saved, React **resumes** exactly where it left off.

Step 7: The Double-Buffer Swap

Once workInProgress is null, the entire WIP tree is built. React enters the Commit Phase (Sync), swaps root.current = WIP, and the changes appear on the screen.

Expert Conclusion

You now have the full chain. React is no longer a "black box" that magically updates. It is a sophisticated task orchestrator that treats your UI tree as a virtual instruction set, executing it in chunks while constantly checking with the browser's hardware clock to ensure your mouse and keyboard feel instant.
